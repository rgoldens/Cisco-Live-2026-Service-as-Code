LTRATO-1001 | Service Insertion as Code: From Concept to Commit
Running the Playbook — What Happens Under the Hood
Before you run ansible-playbook ce-access-vlan.yml, here's exactly what Ansible does with your variables
ce-access-vlan.yml — your completed vars: block
vars:
  vlan_config:        # ① top-level variable name
    n9k-ce01:         # ② key = device name
      id:   23         # ③ VLAN ID for this device
      name: "CLIENT-VLAN-23"
    n9k-ce02:         # different device, different values
      id:   34
      name: "CLIENT-VLAN-34"
①
vlan_config is a nested dictionary. The outer keys are device names — they must match exactly what's in inventory.yml, including capitalisation and hyphens.
②
Each device name maps to its own set of values. This is how one playbook carries per-device data without any if/else logic.
inventory.yml — where the device names come from
nxos:             # ④ group name — matches "hosts: nxos"
  hosts:
    n9k-ce01:       # ⑤ this becomes inventory_hostname
      ansible_host: 172.20.20.30
    n9k-ce02:       # ⑤ so does this
      ansible_host: 172.20.20.31
④
nxos is the group. In the playbook header hosts: nxos tells Ansible to run this play on every host in this group — both switches.
⑤
n9k-ce01 and n9k-ce02 are the host names. Ansible sets inventory_hostname to whichever name it is currently running a task on.

Why this matters: You wrote {{ vlan_config[inventory_hostname].id }} once — but the same task ran on two devices and produced two different values. That's the power of inventory_hostname: a single playbook scales to 2 devices, or 200, without changing a single task. All you change is the vars: block — one entry per device.
