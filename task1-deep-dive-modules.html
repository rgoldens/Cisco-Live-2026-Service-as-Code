Deep Dive: How NX-OS Modules Work in Ansible
Understanding the naming convention, parameters, and state behavior — a reference you can apply to any NX-OS resource
Module Naming — What does "cisco.nxos.nxos_vlans" actually mean?
cisco
Namespace
The vendor who publishes and maintains this code. Cisco owns the cisco namespace on Ansible Galaxy.
.
nxos
Collection
The platform. A collection is a package of related modules. cisco.nxos contains ~100 modules — one for each NX-OS resource: VLANs, interfaces, BGP, VRFs, ACLs, OSPF, and more.
.
nxos_vlans
Module
The specific resource module. Each module manages exactly one resource type. Swap nxos_vlans for nxos_bgp_global and you're managing BGP — same pattern, different resource.
ansible-galaxy collection install cisco.nxos — run this once to install the collection on any Ansible control node
Key Parameters — What you configure in a task
config: list / elements=dictionary
The list of resources you want to configure. Each item in the list is one resource instance — in this case, one VLAN. You can configure multiple VLANs in a single task by adding more items to the list.
# One task — two VLANs config: - vlan_id: 23 name: CLIENT-VLAN-23 - vlan_id: 34 name: CLIENT-VLAN-34
vlan_id: required
The VLAN number (1–4094). This is the identifier — the module uses it to find the VLAN on the device and compare against desired state. Parameters marked required must always be provided — the task will fail without them.
# Using a variable from vars: vlan_id: "{{ vlan_config[inventory_hostname].id }}"
state: default: merged
Controls the behavior of the module — not the state of the VLAN itself. This is the most important parameter to understand. See the table below for all options. In this playbook we always use merged — the safest choice.
# Top-level state (module behavior) state: merged # safe default
{{ inventory_hostname }}
A magic variable Ansible provides automatically. It always equals the name of the device the current task is running on. When running on n9k-ce01 it resolves to "n9k-ce01" — so vlan_config[inventory_hostname].id looks up that device's entry in your vars.
# Same task — two devices — two values # n9k-ce01 → vlan_config["n9k-ce01"].id → 23 # n9k-ce02 → vlan_config["n9k-ce02"].id → 34
state: — The Four Values You Need to Know
Value What Ansible does When to use it
merged (default) Adds or updates only what you specify. Leaves everything else on the device untouched. First choice for most tasks. Safe to re-run — won't remove anything you didn't tell it to remove.
replaced Fully replaces the config for the specific resources you list. Other resources on the device are left alone. You want to completely redefine a VLAN's config from scratch, not just add to it.
overridden Replaces all resources of this type on the device. Anything not in your task gets deleted. Use with caution. Good for enforcing a complete desired state — removes unmanaged config.
deleted Removes the resources you specify from the device. You want to delete specific VLANs. Without a config list, deletes all VLANs.
You now know the pattern. Every NX-OS module in this lab follows the same structure: cisco.nxos.<resource>, a config: list describing desired state, and state: merged to apply it safely. Task 2 uses nxos_l3_interfaces and nxos_ospf_interfaces. Task 3 uses modules from cisco.iosxr and cisco.ios — different platforms, identical pattern. To find any module: browse docs.ansible.com → Collections → cisco.nxos and search by resource name.
