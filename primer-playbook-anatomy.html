Anatomy of an Ansible Playbook
Every playbook has the same three sections — PLAY, vars, tasks — learn this once and you can read any playbook in the lab
ce-access-vlan.yml
— PLAY
- name: "Configure access VLANs"
hosts: nxos
gather_facts: false
— VARS (data — you edit this)
vars:
vlan_config:
n9k-ce01:
id: 23
name: "CLIENT-VLAN-23"
n9k-ce02:
id: 34
name: "CLIENT-VLAN-34"
— TASKS (logic — already written for you)
tasks:
- name: "Create access VLAN"
cisco.nxos.nxos_vlans:
config:
- vlan_id: {{ vlan_config[inventory_hostname].id }}
name: {{ vlan_config[inventory_hostname].name }}
state: merged
PLAY — Targets a group of devices
One play = one group of devices + a list of tasks. A playbook can contain multiple plays targeting different device groups.
hosts: nxos
Which devices from inventory.yml to configure. The group nxos contains n9k-ce01 and n9k-ce02 — Ansible runs every task on both automatically.
vars: — DATA, you edit this
The values specific to your network — VLAN IDs, names, IP addresses. This is the only section you touch in this lab.
tasks: — LOGIC, already written for you
The automation steps. Each task calls one module. The logic stays the same across all deployments — only the vars change.
cisco.nxos.nxos_vlans — Module
The Ansible plugin that performs the work. namespace.collection.module — three parts that tell Ansible exactly what code to run.
{{ "{{ variable }}" }} — Variable substitution
Double curly braces pull a value from vars at runtime. inventory_hostname is set automatically by Ansible to the current device name.

In this lab, you edit the vars — not the tasks. The tasks: section is already written and tested. Your job is to fill in the correct values in vars: for your network. This mirrors real-world IaC practice: engineers update data, the automation logic stays unchanged across every deployment.
