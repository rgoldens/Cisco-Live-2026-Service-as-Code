Anatomy of an Ansible Playbook
Every playbook has the same three sections — PLAY, vars, tasks — learn this once and you can read any playbook in the lab
ce-access-vlan.yml
PLAY
- name: "Configure access VLANs"
hosts: nxos
gather_facts: false
vars:
vlan_config:
n9k-ce01:
id: 23
name: "CLIENT-VLAN-23"
n9k-ce02:
id: 34
name: "CLIENT-VLAN-34"
tasks:
- name: "Create access VLAN"
cisco.nxos.nxos_vlans:
config:
- vlan_id: {{ vlan_config[inventory_hostname].id }}
name: {{ vlan_config[inventory_hostname].name }}
state: merged
PLAY — Targets a group of devices
One play = one group of devices + a list of tasks to run on them.
hosts: nxos — Which devices from inventory
The group nxos contains n9k-ce01 and n9k-ce02. Ansible runs every task on both automatically.
vars: — DATA, you edit this
VLAN IDs, names, IP addresses — the values specific to your network. The only section you touch in this lab.
tasks: — LOGIC, already written for you
The automation steps. Each task calls one module. Logic stays the same across every deployment.
name: — Human-readable description
Displayed in output so you know which step is running. No functional effect.
cisco.nxos.nxos_vlans — Module (the plugin)
namespace.collection.module — three parts that tell Ansible exactly what code to run on the device.
{{ variable }} — Variable substitution
Double curly braces pull a value from vars at runtime. inventory_hostname resolves to the current device name automatically.

In this lab, you edit the vars — not the tasks. The tasks: section is already written and tested. Your job is to fill in the correct values in vars: for your network. This mirrors real-world IaC practice: engineers update data, the automation logic stays unchanged across every deployment.
